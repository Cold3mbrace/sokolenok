# v51.0 — anti-abuse rate limiting (Заход 1)

Подготовка к запуску в люди. Без этого первый же бот-фермер натворит дел.

## Лимиты

- **Telegram callback**: 5 / 5 мин / IP → нельзя массово создавать аккаунты
- **Создание поста**: 10 / час / юзер → защита от спама в ленте
- **Сообщения (DM)**: 30 / мин / юзер + 15 / мин / пара → быстрее обычного типиста, ловит спам
- **Загрузка файлов**: 20 / час / юзер → диск не выжрут (макс 100 МБ/час/юзер на 5 МБ файлах)
- **Заявки в друзья**: 30 / час / юзер → анти-flood уведомлений
- **Комментарии**: 30 / час / юзер
- **Поиск**: 60 / мин / IP → защита от скрапера индекса
- **Лента (`/api/feed`)**: 120 / мин / IP → самый тяжёлый эндпоинт

## Поведение

- При превышении: HTTP 429, body `{ ok: false, error: 'rate-limited', retry_after: N }`, заголовок `Retry-After: N`
- На фронте: toast «Слишком быстро. Подожди N мин.» — юзер видит что произошло
- Telegram-callback редиректит на лендинг с `?auth=tg-too-many`

## Реализация

- Новый модуль `lib/rate-limit.js` — in-memory Map, sweep по таймеру, без зависимостей
- Single-process PM2 — общий Map хранит счётчики, сбрасывается на рестарт (rate-limit это анти-абуз, не безопасность; перезапуск окей)
- IP берётся из `X-Forwarded-For` (за nginx), fallback на socket-адрес
